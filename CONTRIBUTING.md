The following is a set of guidelines for contributing to DashArchive. These are just guidelines, not rules. Use your best judgment, and feel free to propose changes to this document in a pull request.

## Code of Conduct

This project and everyone participating in it is governed by the [DashArchive Code of Conduct](CODE_OF_CONDUCT.md). By participating, you are expected to uphold this code.

## How Can I Contribute?

### Reporting Bugs

- **Ensure the bug was not already reported** by searching on GitHub under [Issues](https://github.com/roketag33/DashArchive/issues).
- If you're unable to find an open issue addressing the problem, [open a new one](https://github.com/roketag33/DashArchive/issues/new). Be sure to include a **title and clear description**, as much relevant information as possible, and a **code sample** or an **executable test case** demonstrating the expected behavior that is not occurring.

### Suggesting Enhancements

- Open a new issue using the **Feature Request** template.
- Explain why this enhancement would be useful to most DashArchive users.

### Pull Requests

1. Fork the repo and create your branch from `main`.
2. If you've added code that should be tested, add tests.
3. If you've changed APIs, update the documentation.
4. Ensure the test suite passes (`yarn test:all`).
5. Make sure your code lints (`yarn lint`).
6. Format your code with Prettier (`yarn format`).

## Styleguides

### Git Commit Messages

- Use the present tense ("Add feature" not "Added feature")
- Use the imperative mood ("Move cursor to..." not "Moves cursor to...")
- Limit the first line to 72 characters or less
- Reference issues and pull requests liberally after the first line

### JavaScript/TypeScript Styleguide

- All TypeScript must pass the linter and type checker.
- We use Prettier for code formatting.
